# Chinese Voice

## Core Voice

- 溫暖、清楚、像資深工程師在帶讀者理解問題。
- 先讓讀者知道「為什麼要在乎」，再展開概念或做法。
- 優先把抽象技術概念放回實際工作情境、團隊協作或判斷場景。

## What To Borrow From The Provided Chinese Samples

- 問題導向標題與切角，例如「X 是什麼？為什麼重要？」
- 開場先丟一個情境、觀察或張力，再帶進主題。
- 用具體例子讓抽象概念落地。
- 從概念解釋一路推到實務意義，而不是停在定義本身。
- 讓文章不只「解釋知識」，也幫讀者建立判斷力。

## How To Use This Voice In `essay` Mode

- 開頭先給一個問題、現象、誤解或工作場景。
- 先讓讀者感受到張力，再下定義。
- 常用「如果...」、「舉例來說...」、「換個角度看...」這類轉場。
- 每個主段都要回答其中一個問題：
  - 這個概念到底在解什麼問題？
  - 它跟常見做法差在哪？
  - 在實務上怎麼判斷它值不值得用？
- 結尾可以拉回更高一層的工作或職涯意義，但要克制。

## How To Use This Voice In `guide` Mode

- 保留清楚、解釋型的優點，但拿掉雙週報問候、促銷段落、品牌口頭禪。
- 第一段直接講問題與成果，不要暖場。
- 如果文章是寫這個個人網站、作品集或部落格的實作，優先直接說「我的個人網站 / 作品集 / 部落格網頁」，通常會比抽象的「內容站」、「這個 repo」更自然。
- 開場在交代問題後，可以立刻補一小段「我真正要處理的是哪些問題」清單，幫讀者快速看懂文章 promise。
- 如果主題有明顯的時效性、版本差異或產業變化，先交代「現在為什麼要在乎」，再進入教學內容。
- 如果文章同時是系列的一部分，先讓首屏成立為一篇獨立可讀的搜尋入口，再補系列定位；不要一開始就把重心放在「這是一篇 Hub / 總覽 / 地圖」。
- H2 以問題、決策或系統部件命名。
- 如果文章是在解釋自己為什麼這樣設計，適度用第一人稱說明判斷與取捨，通常會比全篇維持抽離敘述更順。
- 條列、命令、檔名、路徑與流程可以直接放上來，讓文章更可查驗。
- 如果文章偏 repo 實作，語氣可以更直接、更密實，不需要刻意加入抒情收尾。
- 如果文章已經有清楚的四層、三層或 checklist 結構，後續段落可以再壓短，避免每一節都重複鋪陳「為什麼重要」。
- 如果一個定義段的資訊已經在前面的條列、標題或程式碼示例裡講清楚了，就不要再用一整段平行句重講一次。
- 如果文章同時是系列的一部分，系列定位可以保留，但不要讓「這是一篇母文／導覽頁」壓過文章本身的主查詢與實際價值。
- 面對大主題時，優先幫讀者排出處理順序，例如「先處理哪一層、再處理哪一層、哪些放到最後微調」。
- 當前生態已有清楚的模組或主流做法時，可以及早亮出推薦起點，降低讀者在抽象原則與實作之間的落差。
- 如果同一層裡其實有不同資源類型或不同 policy，例如 API route、列表頁、圖片資源，就拆開寫，不要全塞在同一節。
- guide 模式可以少量使用短 blockquote（`>`）強調一句規則、決策或關鍵提醒，但只在那句話本身夠有力時才用。
- 若一段只是為了製造語氣上的對比，例如反覆寫「不是 A，而是 B」，通常可以改成更直接的操作描述。
- 收尾可以很短。當正文已經把規則與分層講清楚時，結尾只要收斂成一句判斷即可。
- `relatedLinks.note` 預設寫成「這個外部資源本身會教你什麼」，不要寫成作者對系列規劃的幕後註解。

## Sentence And Rhythm Guidance

- 句子可中等偏長，但每段只承載一個主要推進點。
- 口語感可以有，但不要散。
- 技術名詞可用「中文 + 英文括號」幫助辨識。
- 中文技術 SEO 文章的 `title`、`description`、`seoDescription`、`alt`、`tags`，預設優先用中文主詞；英文術語留給括號、首次出現或必要辨識。
- 條列前最好先有一句總領，讓清單不是硬插進去。
- 反問句可以用，但只在真的能幫助推進時使用。
- 搜尋導向文章的標題、摘要與首段可以更直白，必要時接受重複核心關鍵詞，前提是讀起來仍自然。

## Avoid

- 直接複製樣本中的雙週報開場。
- 為了模仿「有溫度」而塞進太多人生感悟。
- 把英文逐句翻成中文，導致語序僵硬。
- 大量空泛形容詞，例如「非常重要」「非常實用」卻沒證據。
- 沒有例子就直接談大觀念。
- 把「文章架構說明」寫得比真正主題還早、還重。
- 把 `description`、`seoDescription` 或首段寫成「這篇在系列裡負責什麼」，卻沒有先講讀者會得到什麼。
- 為了避免 metadata 重複而故意把 `title`、`seoTitle`、`description` 改成三種不同說法，最後反而失焦。
- 明明是個人網站或作品集實作，卻一路用過度抽象的產品詞，讓文章失去真實場景。
- 每一節都再講一次同樣的設計哲學，讓 guide 從實作文滑回觀念文。
- 已經講清楚的定義還硬拉成對仗或排比句，讓文章變得比實際需要更重。
- 為了「寫得完整」保留太長的總結，讓文章最後又重講一次已經成立的結論。
- `relatedLinks.note` 寫成「這個連結在我的文章裡扮演什麼角色」，而不是說明連出去後能看到什麼。

## Quality Check

- 第一段是否已經講清楚讀者為什麼要看？
- 每個段落是否都有明確推進，而不是同義反覆？
- 抽象概念旁邊是否有實例、對比或實務判斷？
- 讀完後，讀者是否更知道怎麼想，而不只是背到一個名詞？
